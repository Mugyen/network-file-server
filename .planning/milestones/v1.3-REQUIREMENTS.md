# Requirements: Network File Server v1.3 — Productionize Friend Tier

**Defined:** 2026-03-16
**Core Value:** Any device can instantly share files with zero setup — scan QR, drop files, done. Works on LAN or over the internet.

## v1.3 Requirements

Requirements for deploying the relay to Google Cloud Run and making it usable as a friend-tier public service.

### Deployment

- [x] **DEPLOY-01**: Relay runs as a Docker container on Google Cloud Run listening on `$PORT`
- [x] **DEPLOY-02**: `GET /health` endpoint returns 200 with mount count for Cloud Run liveness probes
- [x] **DEPLOY-03**: All relay and agent logging uses structured JSON format (severity + message) to stdout
- [ ] **DEPLOY-04**: Session cookies set `Secure` flag when served behind HTTPS (detected via `X-Forwarded-Proto`)
- [ ] **DEPLOY-05**: Relay CORS allows only explicitly configured origins (no wildcard) with `allow_credentials=True`
- [ ] **DEPLOY-06**: uvicorn starts with `--proxy-headers` so `request.url.scheme` reflects the real protocol

### Abuse Prevention

- [ ] **ABUSE-01**: Mount registration is rate-limited to 5 per hour per IP
- [ ] **ABUSE-02**: Proxy requests to `/m/{code}/*` are rate-limited to 300 per minute per IP
- [ ] **ABUSE-03**: Relay enforces a maximum mount TTL (default 24h, configurable via env var) — agents cannot create indefinite mounts
- [ ] **ABUSE-04**: Maximum concurrent mounts per agent IP is capped (default 5, configurable via env var)
- [ ] **ABUSE-05**: Rate limit violations return HTTP 429 with `Retry-After` header

### Persistence

- [ ] **PERS-01**: Mount metadata (code, agent IP, status, created_at, expires_at) persists in SQLite across relay restarts
- [ ] **PERS-02**: On relay restart, all previously-online mounts are marked offline until their agent reconnects
- [ ] **PERS-03**: Agents reconnecting with a preferred code reclaim their existing mount record
- [ ] **PERS-04**: Expired mounts are cleaned up from SQLite by a background sweep

### Landing Page

- [ ] **LAND-01**: Relay root URL (`/`) shows a landing page with project description and mount code entry form
- [ ] **LAND-02**: Mount code form redirects to `/m/{code}/` on submit
- [ ] **LAND-03**: All relay pages include OpenGraph meta tags (`og:title`, `og:description`, `og:image`) for social link previews
- [ ] **LAND-04**: Landing page links to GitHub repo

### Connection Status

- [ ] **CONN-01**: Web UI shows "Host Offline" overlay when relay returns 503 (agent disconnected)
- [ ] **CONN-02**: Web UI shows "Mount Expired" overlay when relay returns 410 (TTL elapsed)
- [ ] **CONN-03**: Offline/expired overlays replace broken partial UI with a clear full-page message

### Drop Box

- [ ] **DROP-01**: Relay starts a default always-on mount in receive mode on boot, serving a temp directory
- [ ] **DROP-02**: Drop box mount code is configurable via env var (default: `dropbox`)
- [ ] **DROP-03**: Reserved mount codes cannot be claimed by external agents
- [ ] **DROP-04**: Landing page links to the drop box mount for "try without installing" experience

### File TTL

- [ ] **FTTL-01**: User can select a TTL (1h, 6h, 1d, 7d) per file at upload time
- [ ] **FTTL-02**: Default file TTL is 1 day
- [ ] **FTTL-03**: Expired files are auto-deleted by a background sweep while the mount is active
- [ ] **FTTL-04**: Connected browsers receive a WebSocket toast notification when a file is auto-deleted
- [ ] **FTTL-05**: File listing shows an expiry badge on files with TTL
- [ ] **FTTL-06**: On mount restart, user is prompted whether to keep or delete files with expired TTLs

## Future Requirements

Deferred to v1.4+.

### Public Launch Polish
- **PUB-01**: Publish `wifi-file-server` and `wifi-relay` to PyPI
- **PUB-02**: Content-Security-Policy headers on all relay responses
- **PUB-03**: Security audit — path traversal fuzz testing, symlink following, header injection

### Scaling
- **SCALE-01**: Multi-instance Cloud Run with distributed rate limiting (Redis)
- **SCALE-02**: Cloud SQL or external database for mount registry

### User Accounts
- **ACCT-01**: Per-user accounts with registration and login on relay
- **ACCT-02**: Per-user mount quotas and bandwidth limits

## Out of Scope

| Feature | Reason |
|---------|--------|
| E2E encryption | Requires browser-side crypto, breaks relay header rewriting — v2+ |
| Custom domains per mount | CNAME + wildcard TLS provisioning transforms relay into PaaS |
| Redis for rate limiting | Unnecessary for single-instance relay; revisit if scaling past 1 instance |
| Multi-instance Cloud Run | Relay is inherently stateful (WebSocket connections); `--max-instances=1` is correct |
| Per-user accounts on relay | Separate milestone (v1.4+) |
| Bandwidth metering / caps | Request rate limits are sufficient for friends tier |
| SameSite=Strict cookies | Breaks cross-site navigation to `/m/{code}/`; Lax is correct |
| Server-side file caching | Pure proxy model per PROJECT.md |
