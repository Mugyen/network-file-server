# Roadmap: v1.3 Productionize Friend Tier

## Overview

v1.3 takes the fully-functional v1.2 relay and makes it safe to run as a public service on Google Cloud Run. Four phases deliver in dependency order: first the container and security fixes that unblock production auth, then rate limiting to protect the public surface, then SQLite persistence so mount codes survive restarts, and finally the UX polish layer (landing page, connection status, drop box, per-file TTL) that depends on all three foundations.

## Phases

- [x] **Phase 12: Cloud Run Foundation** - Dockerize the relay, add health check and structured logging, fix HTTPS cookie/CORS/proxy-header security bugs
- [ ] **Phase 13: Abuse Prevention** - Rate-limit mount registration and proxy requests, enforce max TTL and per-IP mount cap
- [ ] **Phase 14: Persistent Mount Registry** - SQLite mount metadata store survives relay restarts; agents reclaim codes on reconnect
- [ ] **Phase 15: UX Polish and Drop Box** - Landing page with OG tags, connection status overlays, always-on drop box, per-file upload TTL

## Phase Details

### Phase 12: Cloud Run Foundation
**Goal**: The relay runs as a deployable Docker container on Cloud Run with all production-blocking security bugs fixed — HTTPS cookies work, CORS is locked down, and proxy headers are forwarded correctly.
**Depends on**: Nothing (first phase of v1.3)
**Requirements**: DEPLOY-01, DEPLOY-02, DEPLOY-03, DEPLOY-04, DEPLOY-05, DEPLOY-06
**Success Criteria** (what must be TRUE):
  1. Running `docker build` and `docker run -e PORT=8080` starts the relay and `GET /health` returns 200 with a mount count field
  2. All relay and agent log output is JSON with a `severity` field when `RELAY_ENV=production`, readable in Cloud Logging
  3. After deploying to Cloud Run, session cookies carry the `Secure` flag and browser requests are not silently stripped of credentials
  4. CORS preflight requests from an unlisted origin are rejected; only origins in `RELAY_ALLOWED_ORIGINS` are allowed
  5. `request.url.scheme` inside the container reflects `https` when Cloud Run's load balancer terminates TLS

### Phase 13: Abuse Prevention
**Goal**: External agents cannot exhaust relay resources — mount registration and proxy traffic are rate-limited, mounts cannot run indefinitely, and no single IP can hold excessive concurrent mounts.
**Depends on**: Phase 12
**Requirements**: ABUSE-01, ABUSE-02, ABUSE-03, ABUSE-04, ABUSE-05
**Success Criteria** (what must be TRUE):
  1. Sending more than 5 mount registration requests from the same IP within an hour returns HTTP 429 with a `Retry-After` header on the 6th request
  2. Sending more than 300 proxy requests per minute from the same IP to `/m/{code}/*` returns HTTP 429 with a `Retry-After` header
  3. An agent cannot create a mount with TTL exceeding 24 hours (or the relay-configured maximum); the relay enforces a cap regardless of what the agent requests
  4. An agent IP already holding 5 active mounts is rejected when attempting to register a 6th

### Phase 14: Persistent Mount Registry
**Goal**: Mount codes and their metadata survive relay restarts — agents reconnect and reclaim their existing codes, and expired mounts are cleaned up automatically.
**Depends on**: Phase 12
**Requirements**: PERS-01, PERS-02, PERS-03, PERS-04
**Success Criteria** (what must be TRUE):
  1. After restarting the relay, previously-registered mount codes still exist in the registry but are shown as offline until their agent reconnects
  2. An agent that reconnects with a preferred code (one it previously held) reclaims its existing mount record rather than creating a new one
  3. Expired mount records are removed from SQLite automatically without requiring a relay restart
  4. A cold-start relay with an existing SQLite file at startup correctly loads all non-expired mount metadata from disk

### Phase 15: UX Polish and Drop Box
**Goal**: The relay has a production-ready public face — a landing page with social previews, clear agent-offline indicators in the SPA, an always-on public drop box for trying without installing, and per-file upload TTLs with auto-deletion.
**Depends on**: Phase 14
**Requirements**: LAND-01, LAND-02, LAND-03, LAND-04, CONN-01, CONN-02, CONN-03, DROP-01, DROP-02, DROP-03, DROP-04, FTTL-01, FTTL-02, FTTL-03, FTTL-04, FTTL-05, FTTL-06
**Success Criteria** (what must be TRUE):
  1. Visiting the relay root URL shows a landing page with a mount code entry form, a link to the drop box, and a link to the GitHub repo; pasting a valid code and submitting redirects to `/m/{code}/`
  2. Sharing the relay URL on a social platform shows a rich link preview with title, description, and image (verified via an OG tag debugger)
  3. When the agent for a mounted drive goes offline, the SPA replaces the file browser with a "Host Offline" full-page overlay; when the mount TTL has elapsed, the SPA shows a "Mount Expired" overlay instead
  4. Visiting the drop box mount URL (`/m/dropbox/` by default) shows the file browser in receive mode without requiring any agent to be running locally; files uploaded there appear instantly
  5. An agent trying to register the reserved `dropbox` code is rejected; the code can only be used by the relay's internal drop box
  6. Uploading a file with a 1-hour TTL shows an expiry badge on the file in the listing; when the TTL elapses, the file is deleted and a toast notification appears in all connected browsers; on mount restart the user is prompted to keep or delete files with expired TTLs
