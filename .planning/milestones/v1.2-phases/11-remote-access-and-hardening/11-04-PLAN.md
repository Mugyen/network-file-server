<objective>
Fix large file uploads through the relay proxy by streaming request bodies as chunked DATA frames instead of embedding them in the OPEN frame metadata.

Purpose: UAT Test 5 (SPA File Browsing Through Relay) fails for uploads >64KB because the relay proxy stuffs the entire request body into the OPEN frame metadata, exceeding the 64KB MAX_PAYLOAD_BYTES frame limit. The response direction already streams correctly; the request direction needs the same treatment.

Output: Working file uploads of any size through the relay tunnel.
</objective>

    **Relay side (relay/app/routers/mount_proxy.py):**

    In `proxy_request`, replace lines 93-112:

    1. Remove `body = await request.body()` entirely.
    2. Remove `"body"` key from the metadata dict.
    3. Add `"content_length"` to metadata: read from `request.headers.get("content-length", "0")` and pass as int. This lets the agent know how many body bytes to expect (0 means no body).
    4. After `await conn.send_open(request_id, metadata)`, stream the request body as DATA frames:
       ```python
       async for chunk in request.stream():
           # Stream in MAX_PAYLOAD_BYTES chunks to stay within frame limit
           offset = 0
           while offset < len(chunk):
               end = offset + MAX_PAYLOAD_BYTES
               await conn.send_data(request_id, chunk[offset:end])
               offset = end
       # Send zero-length DATA frame as end-of-body sentinel
       await conn.send_data(request_id, b"")
       ```
    5. Import MAX_PAYLOAD_BYTES from tunnel.constants (already imported via FIRST_BYTE_TIMEOUT_S line).

    **Agent side (agent/proxy.py):**

    In `handle_open_frame`, replace body extraction (lines 49-64):

    1. Remove `body_str = metadata.get("body", "")` and `content = body_str.encode("latin-1")`.
    2. Before the `asgi_client.stream()` call, read DATA frames from `conn.read_stream_iter(request_id)` to reconstruct the request body. The stream was already opened by `_agent_receive_loop_with_metadata` in `agent/connection.py` (line 107-110 calls `conn.open_stream` is NOT called there -- it's called in mount_proxy.py line 111 on relay side; on agent side `_dispatch_frame` routes DATA frames to the stream queue, but `open_stream` is NOT explicitly called for OPEN frames in the agent receive loop). IMPORTANT: The agent side needs `conn.open_stream(request_id)` called before reading DATA frames. Currently `handle_open_frame` does NOT call `open_stream` -- the relay side calls it. Add `conn.open_stream(request_id)` at the start of `handle_open_frame` before reading body DATA frames.
    3. Read body chunks until the zero-length sentinel:
       ```python
       body_parts: list[bytes] = []
       async for chunk in conn.read_stream_iter(request_id):
           if len(chunk) == 0:
               break
           body_parts.append(chunk)
       content = b"".join(body_parts)
       ```
    4. IMPORTANT: After reading the body, open a NEW stream for the response. Call `conn.open_stream(request_id)` again? No -- `read_stream_iter` calls `remove_stream` when done. But the response DATA frames from `send_data` need a stream to land on. The solution: do NOT use `read_stream_iter` (which auto-removes the stream on CLOSE signal). Instead, manually read from the stream queue until the zero-length sentinel, then reuse the same stream for the response.

    Actually, the cleaner approach: the zero-length sentinel is just a regular DATA frame with empty payload. After receiving it, stop reading body, but the stream remains open. The subsequent `send_data` calls for the response will also go into this same stream queue, but that's fine because those are being SENT by the agent, not received. The agent SENDS response DATA via `conn.send_data` and the relay READS them via `conn.read_stream_iter`. The stream queues are per-connection-side.

    Revised approach: On the agent side, the stream queue receives DATA frames dispatched by `_dispatch_frame`. The agent needs to:
    1. `conn.open_stream(request_id)` to register the stream
    2. Read DATA frames from `state.queue` manually (not via `read_stream_iter` which auto-removes):
       ```python
       state = conn.get_stream(request_id)
       body_parts: list[bytes] = []
       while True:
           chunk = await state.queue.get()
           if len(chunk) == 0:
               break
           body_parts.append(chunk)
       content = b"".join(body_parts)
       ```
    3. After reading body, proceed with `asgi_client.stream()` and send response as before.
    4. The CLOSE frame at the end calls `conn.send_close()` which sends to relay -- this is fine.
    5. After sending CLOSE, remove the stream: this is already handled by the existing `await conn.send_close(request_id)` on line 85.
